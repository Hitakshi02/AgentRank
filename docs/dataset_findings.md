**YES — confirmed decodable.** From live BigQuery data:

| value (slot0) | score (slot1) | dimensionKey | interpretation |
|---------------|---------------|--------------|----------------|
| 1 | 100 | service_quality | 100/100 quality |
| 1 | 95 | service_quality | 95/100 quality |
| 1 | 88 | responseTime | 88/100 responsiveness |
| 1 | 58 | liveness | 58/100 liveness |
| 1 | 96 | review | 96/100 overall |
| 1 | 99 | service_quality | 99/100 quality |
| 2 | 100 | starrated | 2-unit positive |
| 4 | 100 | starrated | 4-unit positive |

**Encoding conclusion:** `slot1` is the quality score (0–100). Use this as the feedback value for
reputation computation. `slot0` is a multiplier/vote-weight (typically 1). The label "valueDecimals"
in the contract ABI is misleading — in practice it stores the 0–100 score directly.

**For Feature 3 scoring engine:** compute per-agent reputation as the weighted average of `slot1`
values (0–100), weighted by giver credibility and recency. This is REAL on-chain data.

**Data size:** 99.9% of feedback events have `>128 bytes` of data (complex ABI-encoded structs).
All are decodable with the slot layout above.

## Q3 — Agent metadata / domain signals

**YES — rich metadata is on-chain in two places:**

### 1. AgentRegistered events (IdentityRegistry, `0xca52...`, 34,422 events)

Each registration event contains an ABI-encoded `string agentURI` field whose value is:
```
data:application/json;base64,<base64-encoded JSON>
```

Decoded example JSON:
```json
{
  "type": "https://eips.ethereum.org/EIPS/eip-8004#registration-v1",
  "name": "test",
  "description": "only test - This is just an AI test, please do not disturb.",
  "active": true,
  "x402Support": false
}
```

Key fields extractable for every registered agent:
- **`name`** — agent's human-readable name
- **`description`** — agent purpose
- **`x402Support`** (boolean) — whether the agent supports x402 pay-per-request ← on-chain!
- **`active`** (boolean) — live status
- **`type`** URL — classification hint (e.g., `#registration-v1`, `#RAG-evaluation`)

### 2. FeedbackGiven events (ReputationRegistry, `0x6a4a...`)

Each feedback event embeds:
- **`agentURI`** — the live agent endpoint URL (`.well-known/agent.json` discovery format)
- **`dimensionKey`** — the specific quality dimension being rated

### Domain classification verdict

Direct on-chain domain tagging does NOT exist (no single `domain=` field). However:
- The `type` URL in AgentRegistered metadata contains domain hints (e.g., `#RAG-evaluation`)
- The `agentURI` in feedback events points to `.well-known/agent.json` endpoints which
  contain structured capability metadata (fetchable at demo time for curated agents)
- The `dimensionKey` in feedback ("responseTime", "service_quality") hints at category

**Feasibility for Feature 6:** Partial. We can extract domain for agents that have:
(a) AgentRegistered metadata with a typed `scope`/`capability` URL, OR
(b) A resolvable `agentURI` endpoint. The 34,422 wild agents are generally resolvable.
Domain filtering is feasible with a BigQuery `REGEXP_EXTRACT` on the metadata string.
